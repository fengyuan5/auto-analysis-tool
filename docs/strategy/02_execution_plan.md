# 推进方案

## 1. 先做什么

下一步不要先做大而全设计，先把最小闭环跑通。

建议按这个顺序推进：

1. 先定义最小记录。
   至少包含模型、代码、文档、运行环境、profiling、分析结论、验证结果。
2. 先定义两类输出。
   小闭环输出单模型优化建议；
   大闭环输出 CANN 能力缺口和芯片需求信号。
3. 先打通一条最小流程。
   `recipe -> 运行 -> profiling -> 入库 -> 自动分析 -> 优化建议 -> 回灌验证`
4. 先用少量代表样例试点。
   每个 recipe 仓先挑 2 到 3 个模型，不要一开始全量接。

---

## 2. 重点要细化什么

下一版建议只细化五件事：

1. 最小闭环怎么跑。
2. 数据底座最小字段是什么。
3. 什么信号产生单模型建议，什么信号上升为 CANN/芯片结论。
4. recipe 接入规范是什么。
5. 优化前后怎么验证收益。

---

## 3. recipe 盘点怎么做

第一轮先看四个仓：

1. `cann-recipes-infer`
2. `cann-recipes-train`
3. `cann-recipes-spatial-intelligence`
4. `cann-recipes-embodied-intelligence`

每个仓先选 2 到 3 个代表样例，不全量展开。

每个样例先只看八件事：

1. 模型是什么。
2. 任务类型是什么。
3. 文档是否完整。
4. 代码入口是否明确。
5. 运行方式是否清楚。
6. 是否容易采 profiling。
7. 是否已有优化说明。
8. 当前最明显缺什么。

盘点后先输出两类结果：

1. 能直接进入第一批试点的样例。
2. 还不能进入试点、需要补齐的样例。

---

## 4. 对 recipe 仓的判断

这几个 `cann-recipes-*` 仓，适合做第一阶段输入源，但不能直接当成现成数据库。

它们的价值是：

1. 已经有一部分模型、代码、文档和运行入口。
2. 适合作为 profiling 采集起点。
3. 适合作为优化建议回灌和复验载体。

但要做成平台输入源，通常还需要补齐。

---

## 5. recipe 需要补齐什么

至少要补齐五类东西：

1. 统一元数据。
   模型名称、任务类型、框架、硬件、CANN 版本、输入规格。
2. 统一运行入口。
   怎么启动、依赖什么、输入是什么、结果输出到哪里。
3. 统一 profiling 产物。
   采哪些指标、文件放哪里、怎么和版本关联。
4. 统一优化记录。
   做过什么优化、收益多少、风险是什么。
5. 统一验证口径。
   输入规模、batch、卡数、环境版本、收益计算方式要可比。

---

## 6. 当前阶段最关键的判断

当前先证明三件事就够了：

1. recipe 能不能稳定接入。
2. 小闭环能不能在代表模型上跑通。
3. 小闭环里的结论能不能逐步沉淀为 CANN 和芯片输入。

这三件事成立，后面的数据库、分析引擎和平台化才值得继续往下做。
